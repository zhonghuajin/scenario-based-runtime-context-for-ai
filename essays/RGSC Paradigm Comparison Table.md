
| Dimension | Context Engineering | Spec-Driven Development (SDD) | TDD-Driven AI Programming | Harness Engineering | RAG + Documentation Curation | **RGSC (Runtime-Grounded Scenario Context)** |
|---|---|---|---|---|---|---|
| **Nature of Context** | Meta-discipline: manages all context types (memory, tool outputs, structure, etc.) | Prescriptive static text (human-authored requirement specifications) | Executable behavioral contracts (test cases define success criteria) | Environmental constraints (rules, linters, CI, documentation boundaries) | Retrieved static knowledge (code snippets and docs matched by semantic similarity) | **Empirical execution evidence (actual execution paths, variable states, inter-thread synchronization dependencies)** |
| **Context Source** | Multi-source: documents, tools, memory, human input | Human-authored specification documents | Human- or AI-written test cases | Human-configured agent runtime environment | Static knowledge bases (codebases, documentation repositories) | **Real runtime records from production systems** |
| **Primary Problem Domain** | General AI interaction quality improvement | Forward engineering: building new software from scratch | Forward engineering: test-driven code generation | Forward engineering: constraining AI agent code generation behavior | Forward + reverse: providing knowledge support for both generation and understanding | **Reverse engineering: understanding, debugging, auditing, and refactoring existing systems** |
| **Engineering Direction** | Direction-agnostic (meta-framework) | Forward (Prescriptive) | Forward (Contractual) | Forward (Constraining) | Bidirectional but primarily static | **Primarily reverse (Empirical), bridgeable to forward (scenario expansion)** |
| **Relationship to Runtime Facts** | No inherent requirement for runtime data | Operates entirely in the static, pre-execution phase | Uses runtime feedback, but only as a **binary signal** (pass/fail) | May include partial dynamic signals (CI results), but as external feedback loops | Retrieves from static knowledge bases by default | **Runtime evidence is the primary and irreplaceable input** |
| **Granularity of Runtime Information Provided to LLM** | No specific requirement | None | Coarse-grained (test results only) | Coarse-grained (CI/linter output) | None (unless knowledge base contains logs) | **Fine-grained (code-block-level execution paths, variable values, causal chains)** |
| **Multithreading / Concurrency Handling** | Not specifically addressed | Specs may describe concurrency requirements but do not capture actual interactions | Tests can verify concurrent outcomes but cannot expose causal structure | Does not model concurrency of the target system | Not specifically addressed | **Core focus: captures actual synchronization dependencies (happens-before), decomposes multithreaded problems into single-threaded causal tracing subproblems** |
| **Noise Control Strategy** | Minimize context, retrieve tools on demand | Specs are inherently low-noise (but may drift from actual implementation) | Test results are low-noise (but carry minimal information) | Reduces agent search space through constraints | Depends on retrieval quality; may introduce irrelevant content | **Aggressive noise elimination: scenario-boundary pruning + temporal/spatial deduplication + retains only actually-executed code paths** |
| **Ability to Answer "What Did the Code Actually Do?"** | Depends on specific implementation | Weak (answers "what the code *should* do") | Weak (answers "whether the code result is *correct*") | Weak (focuses on agent behavior, not target system behavior) | Moderate (can retrieve relevant knowledge, but cannot confirm actual execution) | **Strong (provides deterministic execution facts directly)** |
| **Academic Validation** | Industry consensus; no single paper | GitHub Spec Kit and industry practices | Extensive academic research; widespread industry adoption | OpenAI Codex team practices | Extensive academic research (RAG surveys, etc.) | **TraceCoder (ICSE '26): runtime tracing yields 34.43% relative improvement; ablation shows accuracy drops from 89% to 78% without runtime info. Cross-validated by CRANE-LLM, InspectCoder, and others** |
| **Core Strength** | High generality; applicable to all AI interaction scenarios | High generation quality when requirements are clear; docs as single source of truth | Fast feedback loops; prevents AI over-engineering | Deterministic constraints improve agent reliability | Reusable enterprise knowledge assets; extends LLM knowledge boundaries | **Causal reasoning instead of guessing; multithreading tractable; zero-noise high signal-to-noise ratio; scenario boundaries naturally bound context size** |
| **Core Limitation** | Too abstract; requires concrete paradigms to operationalize | Specs may drift from actual implementation; high upfront cost | Provides only binary feedback; cannot localize root causes | Depends on manual configuration; does not address understanding existing systems | Knowledge base quality bottleneck; static information may become stale | **Requires a running system; requires triggerable scenarios; instrumentation coverage determines analysis completeness; not yet validated end-to-end at production scale** |
| **Applicable Phase** | All phases | Design → Implementation | Implementation → Verification | Implementation (agent execution environment) | All phases (knowledge assistance) | **Maintenance → Debugging → Auditing → Refactoring → Scenario Expansion** |
| **Relationship to Other Paradigms** | Meta-framework; subsumes all other paradigms | Combinable with TDD, Harness | Combinable with SDD, Harness | Combinable with SDD, TDD | Combinable with all paradigms | **Complementary layer: serves as runtime context source for Context Engineering; enriches RAG knowledge bases with dynamic behavior; provides runtime data input for Harness-based agent workflows** |
